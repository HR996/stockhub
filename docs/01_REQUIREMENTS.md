# istock v1 需求文档

> 本文档只描述**产品定位、系统能力、系统级约束、边界与非目标**。
> 具体的用户故事与验收标准（EARS 格式）见 `docs/02_USER_STORIES.md`。
> 表结构与字段设计见 `docs/05_DATA_MODEL.md`。
> 板块动量因子的算法规范见 `docs/algorithm.md`（后续将迁入 05 的因子章节）。

---

## 1. 产品定位

**istock v1** 是一个面向非技术用户、可通过网页访问的 A 股量化分析原型系统。

系统维护一份共享的 A 股基础数据，向用户提供：数据健康状态可视化、类似数据库管理工具的数据浏览体验、以股票和行业为中心的数据视图、以及可参数化配置的板块级动量因子分析。

v1 的重点**不是**交易策略，而是先**建立稳定的数据底座、可理解的数据浏览体验、清晰的数据健康状态、以及可复用的板块级动量因子分析流程**，为后续 v2 的策略化能力打好基础。

## 2. 目标用户与使用场景

### 2.1 目标用户

- 少量共享该系统的分析人员（v1 不区分角色，全体用户共用同一套数据与配置空间）
- 用户具备基础的股票市场知识，但不要求具备编程或 SQL 能力

### 2.2 典型使用场景

- **打开系统首先查看数据健康状态**，确认最近交易日行情是否已完整入库
- **浏览系统内的表数据**（股票基础信息、K 线、行业分类等），像使用数据库管理工具一样筛选、排序、隐藏字段
- **点击某支股票**，查看该股票的基础信息、交易状态、行业归属、K 线走势
- **在申万官网下载最新分类文件后上传到系统**，系统解析压缩包并更新分类数据，支持出错时按天回滚
- **在因子分析页配置参数**，计算板块动量得分并保存参数配置以便复用
- **查看异常日志**，判断数据更新失败是否需要人工介入

## 3. v1 系统能力清单

按功能领域列出"系统能做什么"。每一项能力对应到 `02_USER_STORIES.md` 中的一个或多个用户故事。**本节不含 EARS 验收标准**。

### 3.1 用户与访问

- **§3.1.1** 预配置用户名登录：仅在系统预配置用户名列表中的用户可进入系统
- **§3.1.2** 登录状态保持：短时间内刷新页面不需要重新登录
- **§3.1.3** 关键操作日志：查询、上传、回滚、查看日志等行为记录用户名、时间、结果
- **§3.1.4** 敏感操作二次密码验证：对高风险操作要求管理员操作密码

### 3.2 数据基座

- **§3.2.1** 股票基础信息：代码、名称、上市日期、市场、交易状态、是否北交所、是否普通股票、是否停牌等
- **§3.2.2** 交易日历：全 A 股交易日 / 非交易日
- **§3.2.3** 日 K 线：近 3 年，同时保存**不复权、前复权、后复权**三种口径
- **§3.2.4** 证监会行业分类：作为申万分类的补充参考
- **§3.2.5** 最新市值：以最新可得数据为准，供板块因子过滤使用（**不追溯历史当日市值**）
- **§3.2.6** 数据更新任务记录：每次同步任务的开始时间、结束时间、成功/失败状态、异常摘要

### 3.3 数据健康

- **§3.3.1** 状态总览：一屏展示行业分类、股票基础信息、K 线、交易日历的最后更新时间与当前状态
- **§3.3.2** K 线月历：以月历视图展示每个交易日的 K 线完整性状态（完整 / 部分缺失 / 完全缺失 / 异常 / 非交易日）
- **§3.3.3** 单日健康详情：点击月历某天，展示当日应更新股票数、成功数、缺失数、异常数、异常原因摘要
- **§3.3.4** 后台自动更新任务的最近结果与异常摘要面向所有登录用户

### 3.4 数据浏览

- **§3.4.1** 数据表列表：展示所有可浏览的数据表，含中文名称与用途说明
- **§3.4.2** 字段说明：每张表的字段展示中文含义
- **§3.4.3** 表内容浏览：分页、排序、条件筛选
- **§3.4.4** 字段显示控制：显示 / 隐藏字段、拖拽调整字段顺序
- **§3.4.5** 页面状态保存：筛选条件、字段顺序、分页位置随页面历史保存与恢复

### 3.5 股票详情

- **§3.5.1** 从数据表、K 线视图或因子结果中点击股票进入详情页
- **§3.5.2** 展示股票基础信息、交易状态标识（北交所 / 普通股票 / 停牌）
- **§3.5.3** 展示证监会行业分类
- **§3.5.4** 展示申万一级 / 二级 / 三级行业分类
- **§3.5.5** 展示日 K 线图，K 线视图上下文中同时展示行业信息

### 3.6 申万行业分类维护

- **§3.6.1** 下载来源说明：页面展示申万官网下载链接与推荐更新频率
- **§3.6.2** 上传：**支持 .rar 与 .zip 压缩包**；解析压缩包内的 xls / xlsx 文件
- **§3.6.3** 解析对齐 TuShare 语义：产出两类结果 —— 分类树（对齐 TuShare `index_classify`）与成分构成（对齐 TuShare `index_member_all`）
- **§3.6.4** 校验：格式校验、字段校验、行业层级校验、股票代码校验
- **§3.6.5** 预览：写入前展示新增 / 删除 / 变更的行业与股票映射数量
- **§3.6.6** 写入：用户确认后才正式写入
- **§3.6.7** 版本记录：每次成功写入生成一个版本快照，记录用户名、时间、来源文件、影响面
- **§3.6.8** 回滚：支持按天回滚到某个历史版本
- **§3.6.9** 派生数据失效标记：分类回滚后，依赖分类的因子结果、宽表标记为"可能失效"
- **§3.6.10** 上传失败保护：解析或写入失败时旧版本不受影响，用户可看到失败原因

### 3.7 板块动量因子分析

- **§3.7.1** 参数化配置：行业体系（SW / 证监会）、**默认展示层级**（申万 L1/L2/L3；证监会仅 L1）、计算日期、动量窗口、Top 比例、收益方式（simple / log）、评分方法（median_return_score / top_count_score）
- **§3.7.2** 交易日约束：计算日期只能选择交易日；`start_date` 按交易日历回溯，不按自然日
- **§3.7.3** 因子计算：按 `docs/algorithm.md` 中的规范执行；**一次提交同时计算所选行业体系下全部层级**（申万一次计算产出 L1、L2、L3 三张结果集；证监会一次计算产出 L1 一张结果集），不做"按需展开"
- **§3.7.4** 结果展示：默认展示所选层级的板块结果表；每行展示板块代码、板块名称、有效股票数、Top 股票数、上榜密度、中位收益、动量得分、是否小样本；支持排序、筛选，展示对应参数摘要
- **§3.7.5** **板块下钻**：任何非最深层级的板块行支持"查看下一级板块"操作 —— 从 L1 板块下钻查看该 L1 下的 L2 板块动量结果；从 L2 板块下钻查看该 L2 下的 L3 板块动量结果；下钻结果直接取自 §3.7.3 已计算好的数据，不重新计算
- **§3.7.6** **板块股票列表**：任何板块行支持"查看板块股票列表"操作 —— 展示当前分类版本下、属于该板块的所有有效股票，默认按当前因子的**个股收益率降序**排序（对齐 `docs/algorithm.md` 定义的收益率）；每行含股票代码、名称、收益率、是否属于全市场 Top 集合；点击股票跳转个股详情（`§3.5`）
- **§3.7.7** **浏览面包屑**：因子下钻与股票查看形成的浏览链条以面包屑形式展示；用户可点击面包屑任一节点回跳；支持删除单条节点、清空整条链；面包屑仅在**当前会话内**保留，页面刷新后清空
- **§3.7.8** 配置保存：命名保存当前参数配置
- **§3.7.9** 配置管理：加载、重命名、复制、删除已保存配置
- **§3.7.10** 结果失效提示：行情数据、行业分类或依赖数据版本变化后，历史结果标记"可能失效"

### 3.8 异常日志

- **§3.8.1** 查看异常日志属于敏感操作，需管理员操作密码
- **§3.8.2** 支持按时间、任务类型、严重程度筛选
- **§3.8.3** 展示异常摘要、发生时间、影响范围、处理状态

### 3.9 添加用户

- **§3.9.1** 添加用户属于敏感操作，需管理员操作密码
- **§3.9.2** 校验用户名不为空、不重复
- **§3.9.3** 添加成功后新用户名可登录

## 4. 系统级约束

### 4.1 数据源

- **主数据源**：baostock（匿名接入，覆盖股票基础信息、交易日历、日 K 线、复权口径、`tradestatus` 停牌标识、`isST` ST 标识）
- **辅助数据源**：akshare，用于兜底获取**最新市值**（baostock 无直接的最新市值 API）
- **申万分类**：不通过 baostock/akshare 拉取，由用户从申万官网手工下载压缩包上传后由服务端解析

### 4.2 数据留存与口径

- 日 K 线保留**近 3 年**
- 权威日 K 线只保存不复权行情，复权因子独立保存
- 最新基准日前复权作为可重建展示缓存；板块因子按指定日期动态复权
- 市值以**最新可得**为准，v1 不建立历史当日市值

### 4.3 数据更新

- 日常数据更新由**后台定时任务**自动完成，用户不通过普通页面手动触发日常更新
- 每次同步任务写入任务日志表，重复执行不应造成数据污染（幂等）
- 自动更新结果对所有登录用户公开可见

### 4.4 派生数据版本

- 所有派生数据（因子结果、宽表快照）必须记录：来源版本、创建时间、创建用户
- 支持"失效标记"：上游数据变化时下游派生结果被标记而非被删除

### 4.5 API 与前后端边界

- 前端只通过 REST API 获取数据
- 前端**不得**执行 SQL 查询、因子计算、数据清洗
- 后端 API 使用统一响应形状：`{ success: bool, data: any, message: string }`
- 分页、排序、错误码、时间格式全站统一

### 4.6 单文件规模

- 单个源码文件不超过 **500 行**（因子服务与申万解析器需要提前按子模块拆分）

## 5. 边界与非目标（v1 明确不做）

1. 不构建 Electron 桌面端
2. 不区分管理员与普通用户角色
3. 不支持复杂用户权限系统
4. 不支持用户在页面中自行配置数据源
5. 不支持双数据源融合（baostock 主 + akshare 仅用于最新市值兜底，非融合）
6. 不支持个股动量因子（v1 只做板块级）
7. 不支持交易下单
8. 不支持实时行情
9. 不建立**历史当日市值**（只保存最新可得市值）
10. 不要求用户手动触发日常行情更新（由后台完成）
11. 不做移动端适配（v1 仅面向 PC 桌面浏览器）

## 6. 术语表

| 术语 | 定义 |
| --- | --- |
| **板块 / Sector** | 一个具体的行业分类节点（例如"电力设备"），根据分类体系与层级唯一标识 |
| **分类体系 / Classification** | 行业分类的来源体系；v1 支持申万（SW）与证监会（CSRC） |
| **层级 / Level** | 分类树的深度：申万支持 L1 / L2 / L3，证监会支持一级分类 |
| **前复权 / 不复权** | 原始行情持久化；前复权按因子动态计算或使用最新展示缓存 |
| **交易日 / Trading Day** | 交易日历中标记为可交易的日期；因子日期回溯必须基于此 |
| **动量窗口 / Window** | 计算个股与板块收益的交易日区间长度（如 20、60、120） |
| **Top 比例 / top_ratio** | 全市场按收益率排名进入 Top 集合的股票占比 |
| **上榜密度 / top_density** | 板块内属于 Top 集合的股票数 / 板块内有效股票数 |
| **成分构成 / Member** | 分类节点包含的股票列表；对齐 TuShare `index_member_all` 语义 |
| **版本快照 / Version Snapshot** | 一次成功的申万分类上传形成的完整数据版本，作为分类树与成分表的引用键 |
| **失效标记 / Invalidation Flag** | 上游数据变化时下游派生数据被标记为"可能失效"，不删除但提示重算 |
| **敏感操作** | 涉及共享数据变更、系统用户变更或异常信息查看的操作，需管理员操作密码 |
